# Day 08 — Variables, VPC, Probes, Loops

## 🔹 Ansible — Install Packages Using Variables

---
- name: Install packages for Pathnex using variables
  hosts: all
  become: yes

  vars:
    pathnex_packages:
      - tree
      - unzip
      - vim

  tasks:
    - name: Install required packages
      yum:
        name: "{{ pathnex_packages }}"
        state: present


## Purpose :
Variables help avoid hardcoding values and make playbooks reusable.    
vars:
  pathnex_packages:
    - tree
    - unzip
    - vim
Creates a variable named pathnex_packages.

Using Variable
name: "{{ pathnex_packages }}"

Ansible replaces the variable with:
tree
unzip
vim
and installs all packages.

Key Concept
Variables store values that can be reused throughout a playbook




## 🔹 Terraform — Create VPC + Subnet

provider "aws" {
  region = "us-east-1"
}

resource "aws_vpc" "PathnexVPC" {
  cidr_block = "10.10.0.0/16"

  tags = {
    Name = "Pathnex-VPC"
  }
}

resource "aws_subnet" "PathnexSubnet" {
  vpc_id                  = aws_vpc.PathnexVPC.id
  cidr_block              = "10.10.1.0/24"
  map_public_ip_on_launch = true

  tags = {
    Name = "Pathnex-Subnet"
  }
}


## Summary 
What is a VPC?
A VPC (Virtual Private Cloud) is your private network inside AWS.

Key Concepts
VPC = Network
Subnet = Portion of network
CIDR Block = IP Range
Public IP Assignment


## 🔹 Kubernetes — Add Liveness & Readiness Probes

apiVersion: v1
kind: Pod
metadata:
  name: pathnex-probe-pod
spec:
  containers:
    - name: web
      image: nginx
      livenessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 5
      readinessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 3


##Summary:
Why Probes?
Kubernetes needs to know:
Is the application alive?
Is the application ready to receive traffic?

Key Concepts
Liveness Probe = Restart unhealthy containers
Readiness Probe = Control traffic routing
Health Checks

# Conditional Stages

Conditional Stages in Jenkins
Conditional Stage means a stage runs only when a specific condition is true.
Instead of running every stage every time, Jenkins checks the condition first.

## 🔹 Jenkins Pipeline — Conditional Execution
You will learn how to **run stages based on branch or parameter conditions**.

pipeline {
    agent any
    parameters {
        string(name: 'ENVIRONMENT', defaultValue: 'dev', description: 'Deployment Environment')
    }
    environment {
        INSTITUTE_NAME = "Pathnex"
    }
    stages {
        stage('Checkout') {
            steps {
                git branch: 'main', url: 'https://github.com/Pathnex/sample-java-app.git'
            }
        }
        stage('Build') {
            when {
                branch 'main'
            }
            steps {
                sh 'mvn clean compile'
            }
        }
        stage('Deploy') {
            when {
                expression { params.ENVIRONMENT == 'dev' }
            }
            steps {
                sh 'echo "Deploying to $ENVIRONMENT by $INSTITUTE_NAME"'
            }
        }
    }
}

## 🔹 GitLab CI — Conditional Jobs
You will learn how to **run jobs based on variables or branches**.

stages:
  - build
  - deploy

build:
  stage: build
  image: maven:3.8.1-jdk-17
  script:
    - git clone https://github.com/Pathnex/sample-java-app.git
    - cd sample-java-app
    - mvn clean compile
  only:
    - main

deploy:
  stage: deploy
  image: alpine:latest
  script:
    - echo "Deploying to $CI_ENVIRONMENT_NAME by Pathnex"
  only:
    variables:
      - $CI_ENVIRONMENT_NAME == "dev"


##Summary:
What is when in Jenkins?
It is used to run stages only when specified conditions are met.

What is only in GitLab CI?
It restricts jobs to run only on specific branches or variables.
