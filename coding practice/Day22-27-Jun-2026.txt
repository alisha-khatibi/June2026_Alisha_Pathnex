## 🔹 Ansible — Use Handler on Config Change

---
- name: Configure Nginx for Pathnex
  hosts: all
  become: yes

  tasks:
    - name: Copy Nginx config
      copy:
        dest: /etc/nginx/nginx.conf
        content: |
          user nginx;
          worker_processes auto;
      notify: Restart nginx

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted

# Explanation :
copy → Creates or updates the Nginx configuration file.
dest → Destination path of the file.
content → Content written into the file.
notify: Restart nginx → Calls the handler only if the file changes.

If the configuration file is unchanged:

Configuration unchanged
↓
Handler NOT executed

If the file changes:

Configuration updated
↓
Notify Handler
↓
Restart Nginx

# Why Use Handlers?

Without a handler:

Every playbook run
↓
Restart nginx

Unnecessary restarts.

With a handler:

Configuration Changed
↓
Restart nginx

Configuration Unchanged
↓
No restart

Answer:
Handlers execute only when notified by another task, making them ideal for restarting services only after configuration changes.


## 🔹 Terraform — Security Group

resource "aws_security_group" "PathnexSG" {
  name   = "pathnex-sg"
  vpc_id = aws_vpc.PathnexVPC.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "Pathnex-SG"
  }
}


## A Security Group controls inbound and outbound traffic for AWS resources using firewall rules.

## 🔹 Kubernetes — DaemonSet

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: pathnex-daemon
spec:
  selector:
    matchLabels:
      app: ds
  template:
    metadata:
      labels:
        app: ds
    spec:
      containers:
        - name: monitor
          image: busybox
          command: ["sh", "-c", "while true; do echo Pathnex Node Monitor; sleep 20; done"]


## What is a DaemonSet?
A DaemonSet ensures one Pod runs on every node in the cluster.

Example Cluster
Cluster

Node-1
Node-2
Node-3

DaemonSet automatically creates:

Node-1 → Pod

Node-2 → Pod

Node-3 → Pod



## 🔹 Shell Script — Monitor Disk Usage

#!/bin/bash

USAGE=$(df -h / | tail -1 | awk '{print $5}')

echo "Disk Usage: $USAGE"

Why Use This?

Useful for:

Monitoring disk space
Creating alerts
Server health checks

Explanation:
This script extracts the current disk usage percentage of the root filesystem using df, tail, and awk.


# Advanced Triggers

## 🔹 Jenkins Pipeline — Trigger on Git Tag
here learn how to **trigger builds on Git tags** with realistic tags.

pipeline {
    agent any
    environment {
        INSTITUTE_NAME = "Pathnex"
        TEAM = "DevOps"
        ENV = "prod"
    }
    triggers {
        pollSCM('H/15 * * * *')
    }
    stages {
        stage('Checkout') {
            steps {
                git url: 'https://github.com/Pathnex/sample-java-app.git', branch: 'main'
            }
        }
        stage('Build') {
            when {
                buildingTag()
            }
            steps {
                sh 'echo "Building $INSTITUTE_NAME app for $TEAM team in $ENV environment"'
            }
        }
    }
}


##Purpose
= Run the pipeline only when a Git tag is created, typically for releases.
= The buildingTag() condition ensures the Jenkins stage runs only for Git tag builds, which are typically used for production releases.

## 🔹 GitLab CI — Trigger on Tag
here learn how to **trigger CI jobs when a tag is pushed**.

stages:
  - build

variables:
  INSTITUTE_NAME: "Pathnex"
  TEAM: "DevOps"
  ENV: "prod"

build:
  stage: build
  image: maven:3.8.1-jdk-17
  script:
    - echo "Building $INSTITUTE_NAME app for $TEAM team in $ENV environment"
  only:
    - tags


## Why Use Tags?
Release builds
Production deployments
Versioning
Interview Answer

only: tags ensures that the GitLab CI job executes only when a Git tag is pushed, making it ideal for release pipelines.
