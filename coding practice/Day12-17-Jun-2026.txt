# Day 12 — File Content, RDS Skeleton, Rolling Update, Backup

## 🔹 Ansible — Ensure File With Content

---
- name: Create config file for Pathnex
  hosts: all
  become: yes

  tasks:
    - name: Write content to file
      copy:
        dest: /etc/pathnex.conf
        content: |
          Welcome to Pathnex DevOps Training
          Today is Day 12


Explanation:
The Ansible copy module can create a file and write predefined content into it.
Why Use It?
Create configuration files automatically
Ensure file content is consistent across servers
Avoid manual editing



## 🔹 Terraform — RDS Skeleton

resource "aws_db_instance" "PathnexRDS" {
  allocated_storage      = 20
  engine                 = "mysql"
  instance_class         = "db.t3.micro"
  name                   = "pathnexdb"
  username               = "admin"
  password               = "password123"
  skip_final_snapshot    = true
}


Explanation:
What is RDS?
RDS (Relational Database Service) is AWS managed database service.
Terraform can provision AWS RDS databases by defining storage, engine type, instance class, and credentials.



## 🔹 Kubernetes — Rolling Update Strategy

apiVersion: apps/v1
kind: Deployment
metadata:
  name: pathnex-rolling
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  selector:
    matchLabels:
      app: roll
  template:
    metadata:
      labels:
        app: roll
    spec:
      containers:
        - name: web
          image: nginx

Explanation:
Rolling Updates replace old Pods with new Pods gradually, ensuring zero or minimal downtime.

## 🔹 Shell Script — Backup Directory

#!/bin/bash

SOURCE="/var/log"
DEST="/backup/pathnex-$(date +%F).tar.gz"

tar -czf $DEST $SOURCE
echo "Backup created at: $DEST"


Explanation:
This script creates a compressed backup of a directory using tar and stores it with the current date in the filename.




# Multi-branch Pipelines

## 🔹 Jenkins Pipeline — Multi-branch
here learn how to **handle multiple branches in Jenkins pipelines**.

pipeline {
    agent any
    environment {
        INSTITUTE_NAME = "Pathnex"
    }
    stages {
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        stage('Build') {
            steps {
                sh 'mvn clean compile'
            }
        }
    }
}

Explanation:
A Jenkins Multibranch Pipeline automatically discovers and builds branches using the Jenkinsfile present in each branch.

Benefits:
Separate builds for each branch
Easier testing of feature branches
Supports Git workflows


## 🔹 GitLab CI — Branch Specific Jobs
here learn how to **run jobs for specific branches in GitLab CI**.

stages:
  - build

build-main:
  stage: build
  image: maven:3.8.1-jdk-17
  script:
    - mvn clean compile
  only:
    - main

build-dev:
  stage: build
  image: maven:3.8.1-jdk-17
  script:
    - mvn clean compile
  only:
    - dev


Explanation:
GitLab CI can run different jobs based on branch names using the only keyword.

Benefits:
Different jobs for different environments
Better CI/CD control
Environment-specific builds
