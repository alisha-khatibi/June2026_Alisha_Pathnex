# Day 18 — Facts, Variables, Affinity, Log Archiving

## 🔹 Ansible — Use Ansible Facts

---
- name: Show system facts for Pathnex
  hosts: all

  tasks:
    - name: Print OS info
      debug:
        var: ansible_facts['os_family']

    - name: Print IP address
      debug:
        var: ansible_facts['default_ipv4']['address']

##How it works:

debug displays information on the screen.
ansible_facts['os_family'] retrieves the operating system family.

Example output:
ok: [server1] =>
ansible_facts['os_family']: RedHat

Why use Ansible Facts?
Execute tasks based on OS.
Get server information automatically.
Avoid hardcoding values.


## 🔹 Terraform — Variables, Outputs, tfvars

# variables.tf
variable "instance_type" {
  default = "c6a.12xlarge"
}

# main.tf
resource "aws_instance" "PathnexServer" {
  ami           = "ami-0abcd1234abcd1234"
  instance_type = var.instance_type
}

# output
output "pathnex_ip" {
  value = aws_instance.PathnexServer.public_ip
}

#Explanation:
tfvars
A .tfvars file stores variable values separately.
Example:
instance_type = "t2.micro"
Instead of editing your Terraform code, you simply change the .tfvars file.

## 🔹 Kubernetes — Node Affinity

apiVersion: v1
kind: Pod
metadata:
  name: pathnex-affinity
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd
  containers:
    - name: web
      image: nginx

Explanation:

requiredDuringSchedulingIgnoredDuringExecution
requiredDuringScheduling → Pod must be scheduled on a matching node.
IgnoredDuringExecution → If the label changes later, Kubernetes does not move the running Pod.
Why use Node Affinity?
Run databases on SSD nodes.
Use GPU nodes for AI workloads.
Control Pod placement.



## 🔹 Shell Script — Archive Log Files

#!/bin/bash

#Explanation:

# Rollback Simulation

## 🔹 Jenkins Pipeline — Rollback on Failure
here learn how to **simulate rollback if deployment fails**.

pipeline {
    agent any
    environment {
        INSTITUTE_NAME = "Pathnex"
    }
    stages {
        stage('Deploy') {
            steps {
                script {
                    try {
                        sh 'echo "Deploying $INSTITUTE_NAME application..."'
                        sh 'exit 1' // simulate failure
                    } catch (Exception e) {
                        echo "Deployment failed. Rolling back $INSTITUTE_NAME application..."
                    }
                }
            }
        }
    }
}


#Purpose
Simulate what happens if deployment fails.

## 🔹 GitLab CI — Rollback Simulation
here learn how to **simulate rollback using after_script**.

stages:
  - deploy

variables:
  INSTITUTE_NAME: "Pathnex"

deploy:
  stage: deploy
  image: alpine:latest
  script:
    - echo "Deploying $INSTITUTE_NAME application..."
    - exit 1
  after_script:
    - echo "Deployment failed. Rolling back $INSTITUTE_NAME application..."

#Explnation:
GitLab CI uses after_script to execute rollback or cleanup steps after a job finishes, even if it fails.
